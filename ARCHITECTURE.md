# ARCHITECTURE — Bazinger PSD → HTML

How this project is wired: every file in the root (the verification/extraction
scripts live in `tools/`), what it does, who/what writes it, when to edit it, and
how the build-and-verify workflow runs. Written for a developer new to the repo.

## What this project is

Rebuild a Photoshop design as a pixel-accurate website. The hard part isn't the
HTML/CSS — it's (1) knowing the exact design values and (2) proving the build
actually matches the design. Every "extra" file exists to solve one of those two
problems. Mental model: the PSD is raw material, the website is the product, and
everything else is measuring equipment and quality control.

## The one rule that removes most confusion: who writes each file

Files fall into three buckets. Mixing them up is the usual source of confusion.

- **Hand-written** (edit directly): `index.html`, `css/style.css`,
  `BUILD-SPEC.md`, `checks.json`, `CLAUDE.md`, `README.md`, `CHANGELOG.md`, and
  the tool scripts themselves.
- **Generated by a script** (NEVER hand-edit — edits are wiped on the next run):
  `psd-spec.json`, `diff.png`.
- **Exported from Photoshop** (not code): `bazinger.png`.

Rule of thumb: if you're about to hand-edit `psd-spec.json` or `diff.png`, stop —
change the thing that _produces_ them instead.

## The authority chain (how value conflicts are resolved)

When sources disagree about a value (which happens constantly), there is a strict
order, enforced by `CLAUDE.md`:

> **bazinger.png (rendered pixels) > psd-spec.json (layer data) > BUILD-SPEC.md (human notes)**

Why: what the design _looks like_ is the real target, so the rendered PNG wins.
Raw PSD layer metadata is a starting hint but sometimes lies (font sizes stored
differently than they render, shadow settings that don't map to CSS, the 0.9489
scale). Hand-written notes are convenient but drift. So: **measure the PNG, use
psd-spec as a hint, treat BUILD-SPEC as guidance.**

## File-by-file

### Inputs (source of truth)

- **`<design>.psd`** — the master design. Kept locally, NOT committed to the repo.
- **`bazinger.png`** — the design flattened to one image, 1404px wide (the PSD's
  native pixel size). This is the answer key: the finished page must match it.
  Don't edit; re-export only if the design itself changes.

### Extracted data

- **`tools/extract_psd_spec.py`** — Python (uses `psd-tools`). Opens the PSD, walks
  every layer, pulls name/position/size/text/font/color/opacity/effects, and
  writes it all to `psd-spec.json`. Lets you read the design automatically instead
  of clicking through layers in Photoshop. Edit only to change extraction logic
  (e.g. the 0.9489 fix); then re-run:
  `python tools/extract_psd_spec.py <psd> psd-spec.json`. Dependencies in
  `requirements.txt`.
- **`psd-spec.json`** — the machine-readable dump of every layer. **Output —
  never hand-edit.** Regenerate by re-running the script.

### The spec

- **`BUILD-SPEC.md`** — the human-readable build manual. Distills
  `psd-spec.json` + hard-won human corrections into numbered sections: §3
  canvas/container/units, §4 colors, §5 fonts, then one section per page part
  (§7 header, §8 hero, …). Two notable sections: §2 the "eyeballing caveat" (a
  running log of cases where raw layer data was wrong until corrected against the
  PNG — a scar log), and §18 the verification workflow. Hand-written. Edit when
  your understanding of a design value changes (e.g. §3 for the coordinate switch,
  §7 when the active bar went 2px → 1px). Lowest authority because it's
  hand-maintained and drifts.

### The build (the deliverable)

- **`index.html` + `css/style.css`** — the actual website. Hand-written by
  following BUILD-SPEC and measuring the PNG. Almost all fidelity work lives in
  `style.css`. Supporting assets in `img/` and `font/`.

### Quality control 1 — computed-style audit

- **`checks.json`** — a checklist: entries of "selector → expected computed
  values" (e.g. `.header .menu .active` → font-size 12px / weight 700 / color
  #4BCAFF), plus which HTML file to test and at what viewport width. Also holds a
  `_visual_only` list (things that can't be audited as text — the logo image, an
  SVG icon) and `.todo-unbuilt` placeholders (so an audit "MISSING" doubles as
  your to-do list). Hand-edit often: when selectors change, when expected values
  change, when you add a section.
- **`tools/style-audit.mjs`** — Node grader. Uses Playwright (an invisible automated
  Chrome) to open `index.html`, read each selector's _actual_ computed style, and
  compare it to `checks.json` — printing PASS/FAIL and a count. Catches typos like
  "13px should be 12px" automatically. **Limit: it only checks text properties**
  (font-size, weight, color, alignment) — not layout, spacing, backgrounds, or
  shadows. Edit the script almost never (we only tweaked its tolerance).

### Quality control 2 — visual heatmap

- **`tools/visual-diff.mjs`** — the other Node grader. Renders the full page with
  Playwright, compares it pixel-by-pixel against `bazinger.png`, and writes
  `diff.png`. Catches everything the text audit can't: positions, spacing, sizes,
  colors, missing elements. It renders at the PNG's width, so the build's
  coordinate space and the PNG's must match (the entire 1404-vs-1480 episode). The
  overall "% different" number is meaningless while lower sections are unbuilt —
  read the heatmap regionally, not as a score.
- **`diff.png`** — the red-where-different heatmap. **Output — never edit;
  gitignored; regenerated each run.** Just open it and look.
- Both `.mjs` tools' dependencies (Playwright, pixelmatch, pngjs) live in
  `package.json`, which also defines the `npm run audit` / `npm run diff`
  shortcuts.

### AI rulebook & docs

- **`CLAUDE.md`** — read automatically by Claude Code in this repo; the agent
  follows it as the project's standing rules. Encodes the authority chain, the
  "never assume — measure" habit, the pre-commit verification gate (run both
  graders, fix to zero, then commit), the ≥1110px fidelity threshold, and a short
  description of each file. Key distinction: **BUILD-SPEC tells you _what_ to build
  (design values); CLAUDE.md tells the AI _how_ to work (process and rules).** Edit
  when project rules change (e.g. 1170 → 1110).
- **`README.md`** — public description for anyone visiting the GitHub repo; this
  is where the AI/tooling use is openly disclosed. Hand-written.
- **`CHANGELOG.md`** — optional running log of notable changes. Hand-written.

## The build loop (per section)

1. **Pick a section** (e.g. §8 hero).
2. **Get the values** — read BUILD-SPEC §8, cross-check `psd-spec.json`, and
   measure `bazinger.png` directly. On any conflict, the PNG wins.
3. **Build it** — write the HTML/CSS.
4. **Verify with both nets** — run `tools/style-audit.mjs` and fix every
   font/weight/color FAIL to zero (update that section's `checks.json` entries
   first); run `tools/visual-diff.mjs`, open `diff.png`, and fix the positions/
   spacing/sizes the red points at, plus your own eyeball against the PNG.
5. **Update BUILD-SPEC** if you learned a correction.
6. **Commit** (you write the message) and **push**.

## Quick reference — when do I touch each file?

| File                                 | Edit when                            | Notes                           |
| ------------------------------------ | ------------------------------------ | ------------------------------- |
| `css/style.css`, `index.html`        | constantly                           | the actual work                 |
| `checks.json`                        | selectors/values change, new section | hand-edit                       |
| `BUILD-SPEC.md`                      | a design value is learned/corrected  | hand-edit                       |
| `CLAUDE.md`                          | a project rule changes (rare)        | hand-edit                       |
| `tools/extract_psd_spec.py`                | extraction logic changes             | then re-run it                  |
| `psd-spec.json`                      | never by hand                        | regenerate via the script       |
| `bazinger.png`                       | the design changes                   | re-export from the PSD          |
| `tools/style-audit.mjs`, `tools/visual-diff.mjs` | tool behavior changes (rare)         | hand-edit                       |
| `diff.png`                           | never                                | generated heatmap; just view it |
| `package.json`, `requirements.txt`   | adding a dependency                  | hand-edit                       |

## The whole machine in one line

PSD → (script) → `psd-spec.json` → (you distill) → `BUILD-SPEC.md` → (you build)
→ `index.html` / `css` → (two graders check against) `bazinger.png` → commit.

Everything else is a dependency list or the AI's rulebook.
